# Wbcom Internationalization Delivery Standard

**Status:** Normative. Applies to every Wbcom WordPress plugin/theme that renders
a user-facing string in PHP or JavaScript.
**Version:** 1.0 (2026-07-08). Distilled from the BuddyNext i18n delivery standard
(v1.0) and the Jetonomy JS i18n rule (1.4.3+); reference implementations in Section 9.

This is the single source of truth. Each plugin keeps a synced copy at
`docs/standards/i18n.md` and a one-line pointer in its `CLAUDE.md`.
Do not re-derive the rules per plugin — link here.

Translations are generated externally (`wbcom-i18n-ai` scans `__()` calls → POT →
per-locale `.mo` + `.json`). Our job is the **delivery plumbing**: every user-facing
string must (a) have a PHP `__()` home so the scanner sees it, and (b) reach the
surface that renders it through a path that actually applies the translation.

## 1. The one rule

> **Every user-facing string has a PHP `__()` / `_e()` / `esc_html__()` home, and
> every render path reads the translated value — never a hardcoded literal.**
> A literal is allowed ONLY as a JS fallback that mirrors an injected `__()` value.
> A literal with no `__()` twin is invisible to the scanner and can never be translated.

**Corollary:** a string that renders English on a fully-translated site is a
**delivery bug**, not a translation gap. File it against the plugin, not the translator.

## 2. Delivery matrix — one row per render layer

| Layer | How a string is delivered | The rule you must not break |
|---|---|---|
| **PHP templates / classes** | `__()`, `esc_html_e()`, `_n()` with the plugin text domain. Textdomain loads on `init:10` via `load_plugin_textdomain`. | Never emit a user-facing literal without a translation function. Never resolve a label before `init` (trap 4). |
| **Interactivity Script Modules** (`assets/js/*/store.js`) | Modules **cannot** use `wp_set_script_translations` (no per-module JED). PHP injects `wp_interactivity_state( '<ns>', [ 'i18n' => [ ... ] ] )`; the store reads `state.i18n.<key>` via `t( 'key', 'English fallback' )`. | **Every `t()` key a store reads MUST exist in that namespace's `i18n_*()` injector.** A read-but-not-injected key silently renders the English fallback on every locale. |
| **Interactivity islands seeded per-instance** | Template passes `'t' => [ 'key' => __( ... ) ]` inside `data-wp-context`; the store captures `ctx.t` at init and reads `t( 'key', 'fallback' )`. | Every key the store reads MUST exist in the template's `'t'` array. Keep array and `t()` calls in lockstep (trap 5). |
| **Classic scripts** (frontend + admin) | `wp_set_script_translations( <handle>, '<domain>', <dir>/languages )` makes `wp.i18n.__()` work, **and/or** `wp_localize_script( <handle>, <globalObj>, [ 'i18n' => [ ... ] ] )` with `__()` values. | If the JS reads `globalObj.i18n.<key>`, that key MUST be seeded in the PHP localize array. Ship the `.json` JED files for `wp.i18n.__()`. |
| **Framework-agnostic shared helpers** (dialog, toast, confirm, side-effect modules) | Not Interactivity stores; they cannot read a per-store state. Localize once onto a shell handle and read `window.<shellData>.i18n.<key>` via an `si( key, fallback )` helper. | Never hardcode a user-facing default here. Internal lists a caller cannot reach (e.g. a report-reason list) MUST come from the shell dictionary — it is their only translatable source. |
| **Data sent to another user** (moderation warnings, notifications) | Resolve the string in the **recipient's** locale on the **server**. | Never POST an actor-side English literal that becomes the recipient's notification. |

## 3. The fallback helper is mandatory

Always ship an English fallback so the UI survives a bridge failure (CSP, script
combiners, a missing injector). But the fallback must be **machine-auditable**.

```js
// REQUIRED — greppable key, enables the Section 7 audit.
const t = ( key, fallback ) => ( state.i18n?.[ key ] ) ?? fallback;
label.textContent = t( 'cancel', 'Cancel' );

// Shared helpers read the shell dictionary instead.
label.textContent = si( 'closeMenu', 'Close menu' );
```

```js
// NOT ACCEPTABLE — works at runtime, but the key is not statically extractable,
// so no tool can diff reads against the injector. This is how trap 1 hides.
label.textContent = state.i18n?.cancel || 'Cancel';
label.textContent = ( window.pluginData?.i18n?.closeMenu ) || 'Close menu';
```

The inline `||` form is the reason "read but not injected" bugs survive review:
you cannot enumerate the read set. Standardize on `t()` / `si()` so
`grep -o "t( *'[a-z]*'"` yields the complete read set.

**A second failure mode the `||` form invites: the unguarded read.** Because the
fallback is optional syntax rather than a required argument, `el.textContent =
D.i18n.loadFail;` is valid and silently renders `undefined` when the key is missing.
`t( 'loadFail', 'Could not load' )` makes the fallback structurally mandatory.
Jetonomy currently carries ~60 unguarded reads in `admin.js` and 6 in `header.js`
precisely because the inline form permits them.

Where a plugin already has inline sites at scale, migrate them mechanically to `t()`
before enabling the Section 7b audit — a partial migration produces a partial read set,
which is worse than none because it reports clean.

## 4. Placeholders and plurals

Use numbered tokens server-side with a translator comment, and `.replace()` in JS.
Never concatenate — word order is not universal.

```php
/* translators: 1: trust level, 2: reputation. */
'trustLevelFormat' => __( 'Level %1$d · %2$d rep', 'plugin-domain' ),
```

```js
const fmt = t( 'trustLevelFormat', 'Level %1$d · %2$d rep' );
trustEl.textContent = fmt.replace( '%1$d', level ).replace( '%2$d', rep );
```

- `_n()` for plurals; keep the whole phrase inside the call.
- Never build a sentence from two `__()` fragments plus a variable.
- `%s` / `%d` are fine when there is exactly one placeholder; numbered tokens are
  mandatory the moment there are two, because translators reorder them.

## 5. Traps that caused real bugs

1. **Read-but-not-injected key (dominant bug).** A store calls
   `t( 'viewMoreComments', 'View more comments' )` but the namespace's `i18n_*()`
   injector never defines `viewMoreComments`. The state has no such key, so the
   English fallback always wins — on every locale, silently. Fixed by adding the key
   to the injector; no JS change. *Audit:* diff every `t()` key against its injector.
2. **JS-only literal.** A string that exists ONLY as a JS fallback (never in any PHP
   `__()`) is invisible to the POT scanner, so no translation is ever produced for it.
   Every fallback literal must have a matching `__()` twin.
3. **Persisted-default-label trap.** An admin screen that persists per-item overrides
   (nav tabs, labels, menu titles) must **only persist a label when it is a genuine
   rename**. Persisting the *default* bakes the admin's locale into the option, and the
   frontend then applies that stored English over the translatable `__()` label.
   Store empty otherwise, and migrate existing rows that equal a known default.
4. **Resolving labels before `init:10`.** `__()` before the textdomain loads returns
   English and, on WP 6.7+, logs `_load_textdomain_just_in_time`. Resolve labels at
   render time, not on early hooks (`plugins_loaded`, `init:0`, class constructors).
5. **Context `'t'` array drift.** A store gains a new `t( 'newKey', … )` but the
   template's `'t'` array is not updated → English fallback. Update both together.

## 6. Adding a new frontend string — checklist

- [ ] The string is wrapped in a PHP translation function somewhere (injector, localize
      array, or template `'t'`), so the scanner sees it.
- [ ] If read by an **Interactivity store**: the key is added to the matching `i18n_*()`
      injector (or the island's context `'t'` array), and the JS uses the same English
      text as its `t()` fallback.
- [ ] If read by a **classic script**: the handle has `wp_set_script_translations`, and
      any `globalObj.i18n.<key>` is seeded in `wp_localize_script`.
- [ ] If read by a **shared helper**: the key is in the shell data `i18n` dictionary and
      read via `si()`.
- [ ] `_n()` / placeholder forms keep the whole phrase translatable (no concatenation).
- [ ] Adding a **new store namespace**? Add its `i18n_<ns>()` injector and call it from
      the central `inject_interactivity_i18n()` in the same commit — even if it ships with
      zero strings, so the path exists before the first string is added.

## 7. Enforcement

### 7a. eslint — validates strings that are ALREADY wrapped

> **Reality check (verified against `@wordpress/eslint-plugin` 22.x):**
> `@wordpress/no-unlocalized-strings` **is not a real rule in this package** — it
> exists in neither the `rules/` dir nor any preset. Earlier docs (including
> Jetonomy's `I18N.md`) claimed to configure it; that claim was false. Do not
> prescribe it. The `@wordpress/i18n-*` rules below only validate strings that are
> *already* wrapped in `__()` (text domain, sprintf, translator comments,
> whitespace). **None of them detects a bare literal** — that is 7b's job.

Ship a **focused** `.eslintrc.json` (do NOT `extends` the recommended preset — it drags
in `prettier` + strict `no-undef` that fail on hundreds of non-i18n lines). List only the
i18n rules, declare the plugin's JS globals, and ignore build output:

```json
"rules": {
  "@wordpress/i18n-text-domain":              [ "error", { "allowedTextDomain": "<domain>" } ],
  "@wordpress/valid-sprintf":                 "error",
  "@wordpress/i18n-translator-comments":      "error",
  "@wordpress/i18n-no-collapsible-whitespace":"error",
  "@wordpress/i18n-no-placeholders-only":     "error",
  "@wordpress/i18n-no-variables":             "error",
  "@wordpress/i18n-ellipsis":                 "error",
  "@wordpress/i18n-no-flanking-whitespace":   "error",
  "@wordpress/i18n-hyphenated-range":         "error"
}
```

Run `eslint "assets/**/*.js" "blocks/**/*.js"` (not bare `wp-scripts lint-js`, whose
default config brings prettier). Must exit zero before every commit.

### 7b. Grep gate — catches trap 2 (bare literals), the class eslint CANNOT

Because no eslint rule catches an unwrapped `el.textContent = 'Cancel'`, ship a
grep-based gate (`bin/check-i18n.sh`). It scans `blocks/` + `assets/js` for literal
assignments to UI-bearing props (`textContent|innerText|innerHTML|placeholder|title`)
and literal `alert/confirm/prompt` bodies, skipping any line that reads from an i18n
source (`t('k','…')`, `si('k','…')`, `state.i18n.x`, `wp.i18n.__`). Exit 1 on any hit.
This is the actual mechanism (Jetonomy's real gate is this, not the eslint rule its doc
named). Reference: `wp-career-board/bin/check-i18n.sh`.

### 7c. Contract audit — catches trap 1 (read but not injected)

eslint **cannot** see the PHP↔JS key contract. A key read by `t()` but never defined by
an injector is a *"read but never written"* defect, which is exactly the bug class
`/wp-contract-audit` exists for. Extend that skill with an i18n key class rather than
writing a bespoke script — its existing severity semantics already fit without change:

- Read set: every `t( 'key', … )` / `si( 'key', … )` in `assets/js/**`.
- Write set: every key in `i18n_*()` injectors, `wp_localize_script` `'i18n'` arrays,
  and template `'t' =>` context arrays.
- `read − write` → **error** ("key read but never written anywhere" — fails the build).
- `write − read` → **warning** ("written-never-read" dead key — fails only `--strict`).

Run it with `--pair` for free/pro. The pair *is* the contract scope: Pro stores read the
free plugin's shell dictionary, so auditing either repo alone reports phantom errors on
one side and misses real ones on the other.

As of v1.0 of this standard, `contract-audit.php` does **not** implement this key class;
i18n appears in neither its checkers nor its documented Limits. Adding it is tracked in
Section 10.

### 7d. Ordering

The 7b grep gate is a *precondition* for the 7c contract audit: until every user-facing
string is a `t()` call with an `__()` twin, the read set is incomplete and the audit
under-reports. Land 7a + 7b first (fix every bare literal), then enable 7c.

## 8. How to verify delivery (per surface)

1. Install and activate a full locale:
   `wp language core install de_DE && wp site switch-language de_DE`
2. Load the surface and **read the emitted state, not just the visible text**:
   - Interactivity: `document.getElementById('wp-script-module-data-@wordpress/interactivity').textContent` → the namespace's `i18n` object.
   - Shared helpers: `window.<shellData>.i18n`
   - Classic: `window.<globalObj>.i18n`
3. Read the result:

   | Observation | Meaning | Action |
   |---|---|---|
   | Key present, German value | Delivered + translated | none |
   | Key present, English value | Delivered; awaiting `.mo` regen | plumbing correct, regenerate |
   | **Key absent** | **Delivery bug** | fix the injector |

4. Restore: `wp site switch-language en_US`
5. After wrapping any previously JS-only literal, regenerate translations
   (`wbcom-i18n-ai`) so the new `__()` strings enter the POT / `.mo`.

Visual inspection alone never counts. An English-looking button on a German site is
indistinguishable between "awaiting translation" (fine) and "key absent" (bug); only the
emitted state tells you which.
