# WP Career Board (Free) â€” 1.2.0 Stability & Decoupling Plan

> **Companion plan**: `wp-career-board-pro/plan/1.2.0-stability.md` (Pro side).
> **Sibling plan**: `docs/PLAN-1.2.0.md` (1.2.0 feature roadmap â€” UTM tracking, WPJM CSV export, REST envelope cleanup). This file is **architectural stability**, not features.

---

## Mission

Make Free **Pro-blind**. After 1.2.0, the Free codebase must contain **zero** references to Pro classes, Pro functions, Pro option keys, or `is_plugin_active('wp-career-board-pro/...')`. All cross-plugin coordination flows through documented filters and actions Free fires, which Pro listens to.

Why this matters for the next 5 years:
- Free is bundled with Reign / BuddyX â†’ it ships standalone to community sites that may never install Pro. Every Pro-aware branch in Free is dead code on those installs and a bug-trap on installs where Pro is removed.
- Pro renames a class or function â†’ Free breaks. We hit this risk every WPCS sweep and every namespace refactor.
- Addon vendors and Pro itself should hook into the same documented surface. Today Pro uses internal `class_exists` lookups that aren't a public contract.

The dependency arrow is **Pro â†’ Free, never Free â†’ Pro**.

- New end-user features (those live in `docs/PLAN-1.2.0.md`).

## F-1 â€” Pro-coordination filter API

**Goal:** One file in Free that declares every filter/action Pro is allowed to hook. Each is a documented contract with a `@since` tag. Pro never reaches into Free's internals; Free never names Pro.

**New file:** `core/class-pro-coordination.php`

**Filters Free fires (in order of priority for the migration):**

| Filter | Default value | Purpose | Replaces |
|---|---|---|---|
| `wcb_pro_active` | `false` | Whether Pro is installed and active | `class_exists('WCB\Pro\Core\ProPlugin')`, `defined('WCBP_VERSION')` |
| `wcb_pro_licensed` | `false` | Whether Pro license is valid | `function_exists('wcbp_is_licensed')` + call |
| `wcb_module_renders` | `[]` keyed array | Map `slug => HTML string` for Pro modules to inject HTML at known render points (alerts subscribe button, notifications bell, board switcher) | 4Ă— `class_exists('WCB\Pro\Modules\â€¦')` template branches |
| `wcb_board_currency` | `(string) $default` | Currency code for a given board | `function_exists('wcbp_get_board_currency')` + call |
| `wcb_pro_upsell_html` | `(string) $default` | HTML to render for Pro upsell at a named location | 4Ă— upsell branches in admin |
| `wcb_pro_settings_saved_notice` | `null` | Text for "Settings saved" notice triggered by Pro | `$_GET['wcbp_saved']` lookup |
| `wcb_search_active_shortcodes` | `['wcb_*']` array of regex/literal patterns | Free's needs-script detection | hard-coded `[wcbp_` test in `class-plugin.php:354` |

**Actions Free fires:**

| Action | When | Purpose |
|---|---|---|
| `wcb_register_extensions` | `init` priority 20 | Pro registers REST routes, blocks, modules here |
| `wcb_pro_pre_check` | After Free fully booted, once per request | Pro verifies version compatibility and bails early if Free is too old |

**Acceptance:**
- New file under 200 lines, doc-block on every filter/action with `@since 1.2.0` and `@param`/`@return`.
- All filters return their `$default` argument unchanged when Pro is absent (existing Free behaviour preserved).
- Filter names locked-in for the next 5 years â€” added to `docs/HOOKS.md` as the documented coordination surface.

**Effort:** S (~1 day).

## F-3 â€” Free owns `wcb_resume` CPT visibility

**Problem:** Free registers `wcb_resume` as `public => false`. Pro hooks `register_post_type_args` (Pro file `modules/resume/class-resume-module.php:141`) to flip it public at runtime. If Pro is deactivated, single-resume URLs 404 with no migration. The CPT contract is owned by Free but mutated by Pro.

**Solution:**

1. Add a Free setting `wcb_settings.resume_archive_enabled` (bool, default `false`).
2. Free's CPT registration reads the setting:
   ```php
   register_post_type( 'wcb_resume', [
     'public'  => (bool) $settings['resume_archive_enabled'],
     'rewrite' => $public ? [ 'slug' => 'resume', 'with_front' => false ] : false,
     // â€¦existing args
   ] );
   ```
3. Add a UI toggle in the Settings â†’ Listings tab: "Public resume archive".
4. Pro's role becomes: on activation, `update_option('wcb_settings', $merged_with_resume_archive_enabled_true)`. Pro stops modifying CPT args at runtime.
5. Add a `wcb_resume_archive_enabled` filter for site owners who want programmatic control.

**Files:**
- `core/class-cpts.php` (or wherever `wcb_resume` is registered) â€” read setting.
- `admin/class-admin-settings.php` â€” add toggle to Listings tab.
- `core/class-install.php` â€” if upgrading from < 1.2.0 AND Pro is active, set `resume_archive_enabled = true` for back-compat.

**Validation:**
- On a site with Pro active + `resume_archive_enabled = true` â†’ `/resume/{slug}` resolves.
- Toggle off â†’ 404 (and a notice in admin warns this hides public resumes).
- Deactivate Pro â†’ resumes still public if the setting says so. Setting is the source of truth.

**Effort:** S (~half day).

**Dependency:** None â€” independent of F-1 / F-2.

## F-4 â€” `wcb_withdraw_application` ability

**Problem:** `api/endpoints/class-applications-endpoint.php:770` (`withdraw_permissions_check`) gates by `$settings['allow_withdraw']` + ownership only. There's no ability â€” site owners can't grant withdraw permission per-role through the standard mechanism Free uses for every other action.

**Solution:**

1. Register `wcb_withdraw_application` in `core/class-abilities.php`:
   - Default-granted to `wcb_candidate` and `administrator`.
   - Category: `wcb_candidate_actions`.
2. Replace the settings check in `withdraw_permissions_check`:
   ```php
   if ( ! wp_is_authorized( 'wcb_withdraw_application' ) ) {
     return $this->permission_error();
   }
   if ( (int) get_post_meta( $app_id, '_wcb_candidate_id', true ) !== get_current_user_id() ) {
     return $this->permission_error();
   }
   return true;
   ```
3. Migrate the existing `allow_withdraw` setting:
   - On 1.2.0 install/upgrade: read existing `allow_withdraw` value. If `true`, leave the ability default; if `false`, revoke from `wcb_candidate` role (preserving site-owner intent).
   - Show admin notice: "The 'Allow Withdraw' setting was migrated to a per-role ability. Manage it under Users â†’ Roles."
4. Mark the setting deprecated in `class-admin-settings.php` (visible but read-only for one cycle), remove in 1.3.0.

**Files:**
- `core/class-abilities.php` (register ability)
- `core/class-roles.php` (grant ability to candidate role)
- `api/endpoints/class-applications-endpoint.php` (`withdraw_permissions_check`)
- `core/class-install.php` (migration)
- `admin/class-admin-settings.php` (deprecation notice)

**Validation:**
- A site owner can revoke `wcb_withdraw_application` from `wcb_candidate` â†’ withdraw button disappears (the candidate-dashboard already passes `allowWithdraw` from settings â†’ must be updated to read from the ability instead).
- Granting the ability to a custom role works.
- Existing sites' candidates retain withdraw permission through migration.

**Effort:** S (~1 day).

**Dependency:** None.

## F-5 â€” Settings consolidation

**Problem:** Two write paths to settings exist:
- `wcb_settings` via Settings API (`admin/class-admin-settings.php` `register_settings()`)
- `wcb_email_settings`, `wcb_captcha_driver` via `admin-post.php` handlers

Two sanitization paths = two bug surfaces. Bug #7 from the recent fix sweep was a missed sanitization location. This won't be the last.

**Solution:**

1. Migrate all settings under `wcb_settings`:
   - `wcb_settings.emails.*` (current `wcb_email_settings`)
   - `wcb_settings.captcha.*` (current `wcb_captcha_driver`)
2. Settings API becomes the single write path. Drop the `admin-post.php` handlers.
3. Update sanitization:
   - Add `emails` and `captcha` sub-arrays to `tab_fields` in `sanitize()`.
   - Each sub-array gets its own sanitize callback (keeps the file maintainable).
4. Migration in `class-install.php`:
   - Read `wcb_email_settings` â†’ write into `wcb_settings.emails`.
   - Read `wcb_captcha_driver` â†’ write into `wcb_settings.captcha.driver`.
   - Bump `wcb_db_version`.
   - Delete the old options.

**Files:**
- `admin/class-admin-settings.php` (consolidate sanitization, drop admin-post handlers)
- `admin/views/email-settings.php` (point form to Settings API)
- `admin/views/captcha-settings.php` (same)
- `core/class-install.php` (migration)

**Validation:**
- Save email settings â†’ only `wcb_settings` row updated; `wcb_email_settings` row no longer exists.
- Save captcha settings â†’ same.
- After migration, all existing values preserved.
- Pro hooks (`wcb_settings_sanitize`) still receive both old and new keys for one back-compat cycle.

**Effort:** M (~2 days, mostly migration + back-compat testing).

**Dependency:** None.

## F-6 â€” CI coupling-regression check

**Goal:** A grep-based gate that fails the CI build if F-2's gains regress.

**New file:** `bin/check-pro-decoupling.sh`

```bash
#!/usr/bin/env bash
set -euo pipefail
violations=$(grep -rn -E '(wcbp_|WCBP_|WCB\\\\Pro\\\\|wp-career-board-pro)' \
  --include="*.php" --include="*.js" \
  --exclude-dir=vendor --exclude-dir=node_modules \
  --exclude-dir=dist --exclude-dir=build \
  --exclude-dir=audit --exclude-dir=plan --exclude-dir=docs \
  . || true)
if [ -n "$violations" ]; then
  echo "ERROR: Free plugin contains references to Pro. Free must be Pro-blind."
  echo "$violations"
  exit 1
fi
```

Wire into the existing CI (`.github/workflows/ci.yml`) as a required check on every PR.

**Validation:**
- Add the script, intentionally introduce a violation, watch CI fail.
- Remove the violation, watch CI pass.

**Effort:** S (~1 hour).

**Dependency:** F-2 must be complete (otherwise gate blocks every PR).

## Risks & rollbacks

| Risk | Mitigation |
|---|---|
| Pro releases late, leaves Free filters returning defaults â†’ some features look broken | F-1 must guarantee Free renders correctly with all filters returning defaults. No filter call may be load-bearing for a Free-standalone site. |
| Migration corrupts existing settings | F-5 migration writes to new keys before deleting old; if any step errors, old keys remain readable. Add a `wcb_settings_migration_failed` flag and surface in admin. |
| Ability migration revokes withdraw from sites that had `allow_withdraw = true` | Don't touch ability defaults during migration unless `allow_withdraw = false` was explicitly set. |
| CI gate too aggressive | Allow the script to be bypassed in `audit/`, `plan/`, `docs/` (already excluded above). |
